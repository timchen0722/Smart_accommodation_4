# 設計 — 風險管理雙檢視(房東管理 / 房源管理 兩頁簽)

日期:2026-07-23 · 目標:把後台分析頁「🚨 風險管理」分頁從「單頁 + 聚焦下拉」改成
「房東管理 / 房源管理 兩頁簽」連動,可自由切頁簽,且點房東自動跳到房源管理頁簽。

> **修訂(2026-07-23,實作後需求變更):** 導覽從「麵包屑(房東檢視 › 房源檢視)」改為
> **兩個常駐頁簽(🧑‍💼 房東管理 / 📋 房源管理)**。因原生 `st.tabs` 無法由程式自動切換,
> 頁簽以兩顆 primary/secondary 按鈕(綁 `rm_view`)實作,才能同時支援「可自由切頁簽」與
> 「點房東ID 自動跳到房源管理頁簽並鎖定該房東」。直接點「房源管理」頁簽(未選房東)顯示全部房源。
> 對應函式:`_tabs(view)`(取代 `_breadcrumb`)、`_go_listings_tab()`(直接點房源管理頁簽,保留篩選)。
> 其餘連動邏輯(篩選 / 可點ID / LIME / 單筆+批次派信 / 通知紀錄)不變。

## 背景

現況 `render_risk_management()`(`modules/platform_sections.py`)為上一輪「一體化聚焦戰情室」的成果:
頂部聚焦狀態列 + 房源風險名單 + 房東層級彙總(下拉聚焦)+ 派發輔導通知(折疊)。
聚焦以 `rm_focus_host`(含 `FOCUS_ALL` 哨兵)為單一跨區塊共享狀態。

使用者要求改為「以導覽為主軸」的雙檢視:預設看房東排行榜,點房東下鑽到他名下房源,
可回頭;房源檢視是實際派信作業面(勾選批次 + 單筆快速發送)。

## 已確認決策

1. **通知中心完全用新模型取代**:房源檢視即派信介面,拿掉門檻滑桿與 🤖 自動寄送 toggle,
   改為 checkbox 批次(底部浮動列)+ LIME 面板單筆發送。
2. **`notify_center.py` 只加公開函式、不改現有行為**:`render_notify_center` 仍供房東入口 TB4 使用,
   本任務沿用其信件組裝/寄送/紀錄邏輯(`_compose`/`_rule_advice`/`_llm_advice`/`_log`),不破壞房東入口。
3. **房東檢視 = 搜尋結果清單**:上方房東ID模糊查詢框 → 顯示「搜尋結果 N 位房東」→
   可點房東ID的排行榜列表(不放 KPI 卡)。
4. **房源檢視房東ID搜尋 = searchable `st.selectbox`**(原生可打字過濾,只能選既有房東ID;
   Streamlit 無可回填 Python 狀態的原生 HTML datalist)。
5. **舊聚焦模型整個移除**(`FOCUS_ALL`/聚焦 banner/聚焦下拉/`_resolve_focus`),對應舊測試改寫。
6. **房源列用 `st.columns` 自繪**(非 `html_table`),才能塞獨立 checkbox + 可點 ID;顯示筆數設上限(預設 100)。

## 技術事實(已查證)

- Streamlit 1.59.1:支援 `st.button(type="tertiary")`(藍色 link 樣式)、`st.container(key=)` +
  `.st-key-*` scoped CSS(可做 `position:fixed` 浮動列)、searchable `selectbox`(打字過濾)。
- `pa.host_risk_summary(df, cm)` 產出欄位 `host_id, 房源數, 高風險間數, 高風險占比, 平均風險分數, 預估年營收`,
  已依「高風險間數 → 高風險占比」降序排序 —— 直接當排行榜資料。
- `_lime_reasons(listing_id, top)`(現存於 `platform_sections.py`)回傳 `[(中文特徵名, 百分點)]`,搬到新模組沿用。
- `notify_center._notify_df()` 已把 `name/host_name/amenities/description/gap_*` 併入預測母體(組信需要)。
- 執行順序地雷:上方區塊只用 `st.session_state.get()` 讀狀態,不放 widget;
  「全選」「清除」等按鈕用 `on_click` callback 在 rerun 前寫 `rm_sel_{id}`(合法,widget 尚未建立)。
- `st.tabs`/區塊每次全渲染 → 雙檢視以 `rm_view` 分流,一次只渲染一個檢視,無效能代價。

## 導覽狀態(session_state)

| key | 值 | 說明 |
|---|---|---|
| `rm_view` | `"hosts"`(預設) / `"listings"` | 目前檢視 |
| `rm_host_filter` | `HOST_ALL="不限"` 哨兵 / int 房東ID | 房源檢視鎖定的房東;直接當房東ID selectbox 的 key。以 `_host_filter()` resolver 把哨兵正規化為 `None` |
| `rm_expanded_id` | int 房源ID / `None` | 目前展開 LIME 面板的房源;再點同一ID收合 |
| `rm_sel_{id}` | bool | 各房源勾選狀態(僅房源檢視存在) |

`HOST_ALL="不限"` 為房東ID selectbox 的「不限」哨兵值(比照舊 `FOCUS_ALL` 慣例)。
上方區塊只透過 `_host_filter()`(讀 `rm_host_filter`、哨兵→`None`)取值,不放 widget,
避開執行順序地雷;點排行榜房東ID 的 callback 直接把 int 房東ID 寫進 `rm_host_filter`。
若該房東因側欄篩選掉出選項清單 → 於 selectbox 建立前重置為哨兵(合法,比照舊 `_render_host_summary` 做法)。

麵包屑:`房東檢視 › 房源檢視`,tertiary link 按鈕。點「房東檢視」→ `rm_view="hosts"` 且清 `rm_host_filter`。
房東檢視時只顯示「房東檢視」單節。

## 房東檢視(landing,無 checkbox、無浮動列)

```
[ 房東ID 模糊查詢:____________ ]        ← st.text_input,子字串比對 str(host_id)
搜尋結果:12 位房東 · 依「高風險間數 → 占比」排序
排名  房東ID(藍底線可點)  房源數  🔴高風險間數  高風險占比  平均風險分數  預估年營收
 1    #12345 ▸            8      6           75%        71%        $1,240 萬
```

- 資料 = `host_risk_summary()`;模糊查詢空字串 = 全部;顯示上限前 100 位(避免渲染過重),
  查詢後在清單上/下方顯示符合房東數。
- 每列房東ID = tertiary link 按鈕,`on_click`:`rm_host_filter=該房東`、`rm_view="listings"`、
  重置 `rm_expanded_id=None`、清空所有 `rm_sel_*`。

## 房源檢視

```
房東檢視 › 房源檢視                                          ← 麵包屑,可點回
篩選:警報層級[🔴]  顯示筆數[100]  風險分數區間[0–1]  房東ID[selectbox:不限/…]
      當房東ID≠不限 → 本行多一顆「☑ 全選目前篩選結果」link
符合 34 間(🎯 已鎖定房東 #12345),顯示風險分數最高的 34 間
─ 每列 ─
☐   #67890 ▸   中山區  整棟  $2,800  🔴 82%  空屋率 61%  房東#12345
      (點 #67890)展開 LIME 面板:
        ▸ 每晚價格 — 推高空屋風險 +3.20 個百分點
        ▸ …(top 3)
        [ ✉️ 產生此房源輔導通知 ]
```

- 篩選:沿用現有「警報層級 / 顯示筆數 / 風險分數區間」+ 新增「房東ID」searchable selectbox
  (選項 `不限` + 全部房東ID;與 `rm_host_filter` 雙向綁定,key=`rm_host_filter`)。
- **每列 = `st.columns([0.5, 1.4, 其他欄位])`**:第一欄 `st.checkbox(key=f"rm_sel_{id}")`,
  第二欄 `st.button(f"#{id} ▸", type="tertiary")` 切換展開,其餘欄位以 markdown 文字呈現。
  checkbox 與 ID 按鈕獨立,不互相誤觸。
- 展開:點 ID → `rm_expanded_id = (None if 已展開 else id)`;展開時該列正下方渲染 LIME 面板 +
  「✉️ 產生此房源輔導通知」按鈕(單筆:呼叫 notify_center 公開寄送函式,寫入通知紀錄,toast)。
- 「全選目前篩選結果」:`on_click` 把目前篩選後所有房源ID 的 `rm_sel_{id}=True`(僅房東ID≠不限時出現)。

## 底部浮動列(批次,新建;僅房源檢視)

- `st.container(key="rm-batch-bar")` + scoped CSS:`.st-key-rm-batch-bar{position:fixed;left/right 避開側欄;bottom:0;z-index}`。
- **只在勾選數 ≥1 時渲染內容**(房東檢視完全不渲染此 container)。頁面底部加 padding 避免遮擋通知紀錄。
- 內容:`已選 N 間　[✉️ 批次發送 N 筆]　[清除選取]`。
  批次發送 = 對所有 `rm_sel_*=True` 的房源逐筆組信+模擬寄送(沿用 `_rule_advice`/`_compose`/`_log`),
  完成 toast、清空勾選並 rerun。清除選取 = 清空所有 `rm_sel_*`。

## 錯誤處理

- 缺 `data/_predictions.csv` 或篩選為空 → 沿用 `guard_scope()` 提示,直接 return(兩檢視前均先守門)。
- 單筆/批次寄送:LLM 失敗退回規則引擎(沿用現有 try/except 樣式),狀態寫入紀錄。
- `rm_host_filter` 指向的房東因側欄全域篩選掉出母體 → 房源檢視顯示「此房東在目前篩選下無房源」並提供回房東檢視。
- `_lime_reasons` 對無足夠特徵的房源回空 list → 面板顯示「此房源無足夠特徵可解釋」。
