# 智慧旅宿空屋率風險預警平台 v3 — 開發規劃書(討論稿)

> 目的:動工前確認範圍與設計。依據 `doc/` 現有文件(SPEC.md、STREAMLIT_APP_SPEC.md、特徵總覽、Workflow docx)與 `data/` 實際資料盤點撰寫。**待討論項目標記為 ❓**。

## 1. 專案定位

第三方房東服務平台:以台北市 Airbnb 資料訓練雙軌模型(迴歸空屋率 + 高風險分類),對房東提供風險預警、SHAP 歸因與策略沙盒;v3 新增**租客入口**與**跨平台市場資料**(591、租租網 ddroom、Booking.com),形成四頁結構:

| 頁面 | 服務對象 | 核心價值 |
|---|---|---|
| 首頁 | 所有訪客 | 平台導覽 + 台北旅宿市場總覽 |
| 房東入口 | 房東 | 風險儀表板、策略沙盒、SHAP 診斷、競品比較 |
| 租客入口 | 找房者 | 跨 4 平台找房比價(日租/月租) |
| 後台分析 | 管理者/評審 | 模型效能、全域解釋、市場分析 |

## 2. 資料盤點與角色分工

### 2.1 建模主資料(Airbnb / Inside Airbnb)
| 檔案 | 規模 | 用途 |
|---|---|---|
| `listings_cleaned.csv.gz` | 6,241 筆 × 81 欄 | 原始房源;推論時撈同區同房型重算百分位 |
| `reviews_cleaned.csv.gz` | 210,288 筆評論 | NLP 情感特徵、房東入口評論摘要 |
| `dataset_final.csv` | 5,849 筆 × 50 欄 | 特徵工程完成之訓練集(含 Y_vacancy / Y_high_risk) |
| `_model_metrics.json` | — | 誠實評估結果:R²=0.234、AUC=0.721(GroupKFold) |

### 2.2 跨平台市場資料(v3 新增)— 核心用途:**每個 Airbnb 房源 1km 內的跨平台競品比較**
| 檔案 | 規模 | 性質 | 主要欄位 |
|---|---|---|---|
| `591_taipei_..._rooms.csv` | 8,066 筆 | **月租**(套雅房) | 月租金、坪數、樓層、格局、經緯度、16 項設施 0/1 |
| `ddroom_taipei_by_rooms.csv` | 544 筆 | **月租**(共生宅為主) | 月租金、押金、管理費、坪數、最短租期、特色標籤、經緯度 |
| `taipei_rooms_only_v14_....csv` | 824 筆 | **日租**(飯店房型) | 房價、評分、評論、行政區、房型、可住人數、經緯度 |

三份資料皆含經緯度,統一建入 BallTree(haversine)索引。以每個 Airbnb 房源為圓心,計算 1km 內各平台競品之:數量密度、價格帶、設施覆蓋率(該設施在周邊競品的擁有比例)。此競品索引與統計即「跨平台競爭模組」的輸入。

**價格正規化(2026-07-18 決議):統一換算為「每人每晚等效價」**

| 平台 | 公式 | 備註 |
|---|---|---|
| Airbnb / Booking(日租) | 價格 ÷ 可住人數 | Booking 可住人數 N/A 時由房型名稱推估,無法推估則排除 |
| 591 / ddroom(月租) | 月租 ÷ 30 ÷ 可住人數 | 「長租等效參考價」,UI 需標註(未含押金/最短租期);591「2人」字串、ddroom「1~2」區間取上限 |

- **分層比較**:每人單價有規模經濟偏差(大房源天然偏低),落點百分位只與**同容量層**(1–2 / 3–4 / 5+ 人)之競品比較。
- **雙軌呈現**:競品卡片顯示原始價格;落點百分位與智慧建議引擎採每人每晚等效價。

### 2.3 POI 輔助資料
捷運出入口、公車站、超商、餐廳、公園、學校、診所 — 用於房東入口「機能快照」與租客入口「通勤/機能篩選」。

## 3. 四大頁面設計

### 3.1 首頁
- Hero 區:平台標題(漸層字)、一句話價值主張、三個入口卡片(房東/租客/後台)。
- 市場 KPI 卡:Airbnb 房源數、平均空屋率、高風險房源占比、跨平台均價對比(Airbnb 日租 vs Booking 日租 vs 591/ddroom 月租折算)。
- 台北市 12 行政區風險地圖縮圖(點擊導向房東入口)。
- ❓ 首頁是否需要「最新通知/公告」區塊?

### 3.2 房東入口(沿用 v2 主體 + 強化)
沿用既有設計:房東選擇器 → 房源選擇器 → 信心 Badge → SVG 半圓風險儀表板 → 策略沙盒(房價/最低入住天數/回覆速度/描述字數)→ SHAP 雙向條形圖 → AI Top 2 診斷建議 → POI 機能快照。

v3 強化(依 0716 討論定向):
1. **1km 跨平台競品視圖**:以選定房源為圓心,地圖呈現 1km 內 Airbnb 競品(風險著色)+ 591/Booking/ddroom 競品(平台色標);附競品統計卡:各平台數量、價格帶對比(月租折算日租)、我的定價落點百分位。
2. **設施差距分析(Amenity Gap)**:比對該房源設施 vs 1km 內競品的設施覆蓋率,找出「周邊多數競品有、我沒有」的缺口(如 WiFi、電梯、洗衣機、自助入住)。
3. **智慧建議引擎(高風險觸發)**:當模型判定高風險(≥70%,黃燈亦提供精簡版),綜合 SHAP 扣分項 + 跨平台價格落點 + 設施缺口,輸出可執行建議:調降租金至同區競爭價位、增加特定設施、放寬最低入住天數、提升回覆速度、充實文案等,並附預估改善幅度(沙盒重算)。
4. **站內通知中心(模擬)**:高風險房源清單、通知紀錄與已處理狀態(不寄真實 Email)。

### 3.3 租客入口(v3 全新)
- 模式切換:**短租(日)** = Airbnb + Booking;**長租(月)** = 591 + ddroom。
- 篩選:行政區、預算範圍、可住人數/格局、設施(WiFi、電梯、洗衣機…)、離捷運距離。
- 呈現:地圖(pydeck/folium)+ 房源卡片列表(圖片、價格、評分、平台標籤、外部連結)。
- 加值:CP 值排序(價格 vs 評分/機能)、「同預算長短租比較」(月租 vs 日租×30 折算)。
- ❓ 租客入口是否也要給「這間房 CP 值分析」之類的 AI 元素,或純搜尋比價即可?

### 3.4 後台分析
- 模型效能:R²/MSE、AUC/Recall/F1、PR Curve、GroupKFold 各折分數。
- 全域解釋:SHAP summary(全域特徵重要度)、關鍵特徵 vs 空屋率關係圖。
- 資料分析:Y 分佈、行政區×房型空屋率熱度、價格分佈。
- 跨平台市場分析:四平台供給量/價格帶對比、行政區行情圖。
- ❓ 是否加入「前向特徵選擇曲線」(193607 文件之 R² vs 特徵數)作為研究成果展示?

## 4. 模型方案(需拍板 ❗)

文件間有兩套說法,需先確認:

| 項目 | SPEC.md(v2 敘述) | 特徵總覽/實測 metrics(誠實版) |
|---|---|---|
| Y 定義 | `1 - estimated_occupancy_l365d/365` | `availability_365 / 365` |
| 特徵數 | 50(含 POI+NLP 進模型) | 37 核心(POI 實驗性未寫回;dataset_final 為 50 欄含實驗欄) |
| R² | 0.5871 | **0.234**(GroupKFold, _model_metrics.json) |
| AUC | 0.9071 | **0.721** |

**建議**:以 `dataset_final.csv` + 誠實 GroupKFold 指標為準(R²≈0.23/AUC≈0.72),後台分析如實呈現並解釋「為何誠實評估較低」(防房東分組洩漏),這是報告亮點而非弱點。SPEC 的高分數不再對外宣稱。

- 演算法:HistGradientBoosting ×2(A 迴歸、B 分類 + Isotonic 校準),GroupKFold(host_id, 5 折),SHAP TreeExplainer。

### 4.1 可抽換功能模組架構(0716 討論新增 ❗)

每個功能模組獨立訓練/建置、獨立序列化為一個 pkl,遵循統一介面,前端只依介面呼叫 — 任一模組可單獨重訓、替換、升級而不動其他模組:

| 模組 pkl | 內容 | 介面 |
|---|---|---|
| `preprocessor.pkl` | 特徵前處理管線 + 同區價格資料庫 | `transform(raw) → X` |
| `regressor.pkl` | 模型 A:空屋率迴歸 | `predict(X) → Y_vacancy` |
| `classifier.pkl` | 模型 B:高風險分類(校準後機率) | `predict_proba(X) → P(high_risk)` |
| `explainer.pkl` | SHAP TreeExplainer | `explain(X) → φ 向量` |
| `competitor_index.pkl` | 跨平台 BallTree 索引 + 競品清理後資料(Airbnb/591/Booking/ddroom) | `query(lat, lon, r=1km) → 競品統計` |
| `suggestion_engine.pkl` | 規則引擎(SHAP 扣分 × 價格落點 × 設施缺口 → 白話建議模板) | `suggest(shap, comp_stats, features) → 建議清單` |

- `train.py` / `build_modules.py` 一鍵重建所有 pkl 至 `models/`;各 pkl 內附 `version`、`built_at`、`schema` metadata,載入時檢查相容性。
- 換城市、換資料源、換演算法時只需重建對應模組。
