# Streamlit 前端網頁與機器學習大腦協同運作機制解析

本文件詳細說明本專案的 **Streamlit 前端網頁介面** 如何在後台呼叫與整合**數據分析與機器學習模型（Model A & B、SHAP 解釋器）**的訓練結果，實現即時預警與經營策略沙盒模擬。

---

## 1. 系統對接架構：後端模型如何「點亮」前端網頁

整個系統的對接流程如下圖所示：

```text
 ┌────────────────────────┐
 │ Streamlit 網頁前端介面 │ 
 └───────────┬────────────┘
             │ (1) 傳遞房源 ID 與房東拉動的沙盒引數 (如房價、天數)
             ▼
 ┌────────────────────────┐
 │    src/inference_api   │ <── 載入 ── models/ 下的 4 個大腦權重檔 (.pkl)
 └───────────┬────────────┘
             │ (2) 撈取同商圈數據，動態重算 price_pctl_nbhd (價格百分位)
             │ (3) 將 50 維特徵向量餵給雙軌模型進行預測與 SHAP 拆解
             ▼
 ┌────────────────────────┐
 │  輸出預測空屋率、高風  │
 │  險機率與 SHAP 貢獻度  │
 └───────────┬────────────┘
             │ (4) 回傳數據，即時渲染網頁組件
             ▼
 ┌────────────────────────┐
 │ 綠黃信心Badge、SVG儀表 │
 │ 板、紅綠SHAP圖、AI診斷  │
 └────────────────────────┘
```

## 2. 前端網頁元件與機器學習大腦的對接細節

以下為你詳細對照 Streamlit 網頁上的各個視覺化元件，分別是使用到哪一個機器學習訓練結果與背後的商務邏輯：

| Streamlit 網頁呈現元件 | 呼叫的機器學習資產 (pkl 檔案) | 背後運作的數據分析與機器學習邏輯 |
| :--- | :--- | :--- |
| **🟢/🟡 預測信心標籤 (Confidence Badges)** | `preprocessor.pkl` (內含房源歷史資料庫) | *   **邏輯**：系統會讀取該房東在平台的歷史紀錄。<br>*   **呈現**：若該房東名下有多個房源或歷史已累積足夠評論，顯示綠色**「預測信心：極高」**；若是新加入、無評論的房源，顯示黃色**「預測信心：中等 (冷啟動)」**。<br>*   *學術意義*：這直接對應我們在 GroupKFold 交叉驗證中發現的「老房東特徵洩漏」現象。 |
| **📊 SVG 半圓形風險儀表板 (Risk Gauge)** | `regressor_model.pkl` (模型 A：迴歸大腦) | *   **邏輯**：當房東點選某間房源時，API 會載入其 50 維特徵，呼叫模型 A 預測出該房源的**連續值年空屋率 $Y$**。<br>*   **呈現**：網頁使用內建的 SVG 繪製出一個半圓弧進度條，指針會動態旋轉到預測的空屋率位置（如 55%）。 |
| **🚨 高風險概率預警與聯動** | `classifier_model.pkl` (模型 B：分類大腦) | *   **邏輯**：如果模型 A 預測的空屋率 $\ge 70\%$，會自動觸發警報，呼叫模型 B（經過 Isotonic 機率校準的分類器），計算**高風險空屋發生的實質機率**。<br>*   **呈現**：儀表板下方會顯示紅字高風險警告，並啟動 CSS 呼吸燈閃爍動畫，顯示「高空屋機率：91.2%」，促使房東立即注意。 |
| **🛠️ 經營策略沙盒 (Strategy Sandbox)** | `preprocessor.pkl` + 雙軌模型 | *   **邏輯**：提供「每晚房價」、「最低入住天數限制」、「客服回覆速度」、「自介文字長度」四個滑桿/輸入框，供房東即時微調模擬。<br>*   **動態百分位重算（關鍵設計）**：**當房東調動房價時，後台 API 會立即拉出同行政區、同房型的所有房源價格，即時重新計算該房源在新定價下的「相對價格百分位 (`price_pctl_nbhd`)」，再將新特徵輸入模型。** 這避免了模型因靜態價格而失真，使滑桿非常靈敏。 |
| **📈 SHAP 雙向橫條歸因圖 (SHAP Waterfall Chart)** | `shap_explainer.pkl` (SHAP 解釋器) | *   **邏輯**：當沙盒參數被調整後，API 呼叫 fitted 的 `TreeExplainer`，將模型 A 的預測結果拆解為 50 個特徵的邊際貢獻度 $\phi_i$。<br>*   **呈現**：網頁以客製化 HTML/CSS 繪製雙向條形圖。以 50% 空屋率為中心：**綠色向左**代表此特徵正在為房源減免空房風險（如價格便宜、離捷運近）；**紅色向右**代表此特徵正在推高空房風險。 |
| **📋 AI 房源智能診斷報告 (Action Plan)** | `shap_explainer.pkl` + 規則引擎 | *   **邏輯**：系統會篩選出當前模擬狀態下，**SHAP 貢獻值大於 0（推高風險最多）的前兩名特徵**。<br>*   **呈現**：將這兩個特徵傳入規則引擎，自動將繁雜的特徵代號轉譯為白話的改善指引（例如：最低入住天數 30 天過長，建議放寬為 1~2 天以吸引短途客）。 |

## 3. 機器學習結果如何動態影響前端：一個具體操作案例說明

為了讓外行人也能懂，我們可以看一個**「房東拉動價格滑桿」**的具體過程：

```text
【房東在網頁上將房價調高 500 元】
       │
       ▼ (Streamlit 監聽到滑桿數值變動)
【呼叫後端 API：run_sandbox_inference】
       │
       ▼ (數據庫重算：得出該房源在同行政區同房型對手中，從原本 50% 便宜變成 80% 偏貴)
【更新特徵向量中的 price_pctl_nbhd = 0.80】
       │
       ▼ (載入 regressor_model.pkl 進行預估)
【模型 A 預測空屋率從 45% 飆升至 72%】
       │
       ▼ (因為 >= 70%，載入 classifier_model.pkl 進行機率估算)
【模型 B 預測高風險發生機率為 82.5%】
       │
       ▼ (載入 shap_explainer.pkl 計算特徵貢獻)
【SHAP 計算出 price_pctl_nbhd 貢獻了 +12.5% 空屋風險，成為 Top 1 扣分項】
       │
       ▼ (將數據與 AI 建議回傳給 Streamlit 前端網頁)
【網頁呈現結果】：
 1. 儀表板變為紅燈，指針轉到 72%。
 2. 顯示高風險機率 82.5% 與呼吸燈警報。
 3. SHAP 橫條圖中，價格百分位出現一條向右延伸的長長紅色扣分條。
 4. AI 診斷書輸出：「您的定價偏高，高於同區 80% 的競爭對手，推高了 12.5% 的空屋風險。建議調降價格。」
```

透過這種設計，機器學習的訓練結果（pkl 檔案）不再只是冰冷的評估指標，而是成為前端網頁**具備即時互動性與商業指導價值的預測核心**！
