# 智慧旅宿空屋率風險預警平台 — 模型與特徵完整說明(X / Y)

> Smart_accommodation_4 · 2026-07-19
> 本文件詳列平台使用的全部資料分析模型、目標變數 Y、59 個特徵欄位 X 之定義與**選用理由**。
> 所有效能數字皆為 GroupKFold(host_id, 5 折)OOF 誠實評估。

---

## 1. 目標變數 Y

| 變數 | 定義 | 選用理由 |
|---|---|---|
| `Y_vacancy`(模型 A 目標) | 未來一年空屋率 = `availability_365 ÷ 365`,值域 [0,1] | 直接對應房東最關心的「房子空著的比例」;取自 Inside Airbnb 日曆的未來一年可訂天數,是**未來供給的真實觀測**,非規則計算值 |
| `Y_high_risk_06`(模型 B 目標) | `Y_vacancy ≥ 0.6` 之二元標籤(基準率 37.4%) | 與平台「60% 通知門檻」業務語意一致;實驗顯示相較 0.7 標籤,0.6 使基準率提高(31%→37%),同門檻下 Recall 由 0.23 提升至 0.27,對通知情境更有利(詳見 doc/01 分析報告) |

**防洩漏鐵律**:凡用於定義 Y 或與入住量同期相關的欄位一律禁入特徵 X,包括
`availability_30/60/90/365`、`estimated_occupancy_l365d`、`estimated_revenue_l365d`、
`number_of_reviews`(及 ltm/l30d/ly)、`reviews_per_month`、`last_review`。
放入這些欄位模型等於「抄答案」,指標虛高但無預測意義。

## 2. 特徵欄位 X(完整模型 59 個;冷啟動變體 52 個)

資料源:Inside Airbnb 台北 listings(6,241 筆)+ reviews(210,288 則)+ 台北市官方 POI 七資料集。
訓練樣本 5,849 筆(剔除經營未滿一年之新手房源,避免累積天數不足造成的空屋率低估偏差)。

### 2.1 結構化硬體(13 個)— 房源的基本盤

| 特徵 | 繁中說明 | 選用理由 |
|---|---|---|
| `accommodates` | 可住人數 | 房源容量的基本刻度,影響客群與定價 |
| `bedrooms` / `beds` | 臥室數 / 床位數 | 前向特徵選擇中 beds 為第 1 步即選入的特徵 |
| `bathrooms_count` / `is_shared_bath` | 衛浴數 / 是否共用衛浴 | 舒適度與私密性,共用衛浴顯著影響訂房意願 |
| `price` | 每晚價格 | 決策核心變數;亦為沙盒 What-if 的操作對象 |
| `minimum_nights` / `maximum_nights` | 最短 / 最長入住晚數 | 訂房彈性;EBM 分析顯示 2~3 晚最優的非線性效應,maximum_nights 長期重要度前 3 |
| `min_nights_avg_ntm` | 近期平均最短入住 | 五大核心訊號之一(前向選擇 ΔR²=+0.035);反映近期經營策略 |
| `instant_bookable` | 可即時預訂 | 降低訂房摩擦,平台演算法加權項 |
| `self_checkin` | 自助入住(由 amenities 關鍵字比對) | Tier1 批量測試驗證有效(R² 0.561→0.572),重要度第 10 |
| `room_type_code` / `neighbourhood_code` | 房型 / 行政區編碼 | 類別基本盤;亦為競爭特徵的分組依據 |

### 2.2 歷史評分(7 個)— 口碑訊號

`review_scores_rating / accuracy / cleanliness / checkin / communication / location / value`
(總體 / 描述準確 / 清潔 / 入住體驗 / 溝通 / 地點 / 性價比評分)

**選用理由**:住客真實口碑是旅客決策關鍵;其中**溝通評分**是前向選擇五大核心訊號之一
(第 7 步 ΔR²=+0.029)。缺失值(約 19% 無評論房源)以同區中位數填補並配合
`has_no_reviews` 旗標,讓模型能區分「新房源」與「低分房源」。

### 2.3 市場競爭(5 個)— 滯銷是「相對」而非「絕對」

| 特徵 | 繁中說明 | 選用理由 |
|---|---|---|
| `price_pctl_nbhd` | 同區同房型價格百分位 | 第三方平台核心設計:貴不貴要跟鄰居比;沙盒調價時動態重排 |
| `score_pctl_nbhd` | 同區評分百分位排名 | 五大核心訊號之一(前向選擇第 11 步即追平 37 特徵基準) |
| `amenities_vs_median` | 設施數 ÷ 同區同房型中位數 | 設施競爭力的相對量度 |
| `nbr_density_1km` / `nbr_density_same_type_1km` | 1km 內房源密度 / 同房型密度 | BallTree(haversine) 計算;供給過剩程度與直接對手數量 |

### 2.4 房東身分與經營品質(7 個)— 最強的一組訊號

| 特徵 | 繁中說明 | 選用理由 |
|---|---|---|
| `host_acceptance_rate` | 房東接受率 | **全模型 SHAP 重要度第一**;接單意願直接決定成交 |
| `host_response_rate` / `response_speed` | 回覆率 / 回覆速度(序數 1~4) | 客服即時性;回覆速度為沙盒可調變數 |
| `host_is_superhost` | 超讚房東 | 平台認證的信任標記 |
| `host_listings_count` / `calculated_host_listings_count` | 名下房源數(自報 / 平台計算) | 專業房東經營規模;亦是冷啟動路由依據(≤1 間走冷啟動模型) |
| `host_tenure_days` | 房東經營天數(last_scraped − host_since) | 經營資歷,重要度長期前 5 |

> 這 7 個特徵在**冷啟動變體中移除**(新房東無歷史),以地點/房間特徵補位——
> 此即雙模型策略的核心設計。

### 2.5 經營用心度(4 個)— 文案與細節

`desc_len`(房源描述字數)、`host_about_len`(自介字數)、
`neighborhood_overview_len`(周邊介紹字數)、`amenities_count`(設施總數)

**選用理由**:描述長度是「經營用心程度」的代理變數,重要度前 10~15;
desc_len 亦為沙盒可調變數,對應「充實文案」的可行動建議。

### 2.6 地點/房間衍生(6 個)— 冷啟動補位組

| 特徵 | 繁中說明 | 選用理由 |
|---|---|---|
| `latitude` / `longitude` | 經緯度 | 緯度為五大核心訊號之一(北投/士林 vs 市中心的地段效應) |
| `hotel_count_1km` / `hotel_count_500m` | 周邊飯店數(OSM) | **反直覺發現**:與空屋率負相關——飯店紮堆處是觀光精華區,為「需求熱度代理」而非競爭懲罰 |
| `airbnb_hotel_supply_ratio` | 短租/飯店供給比 | 供給結構描述 |
| `price_per_person` / `price_per_bedroom` / `beds_per_person` | 每人 / 每臥室單價、每人床數 | 價格正規化,消除容量規模差異(冷啟動模型重要度高) |

### 2.7 地理 POI(11 個)— 台北市官方開放資料

| 特徵 | 繁中說明 | 來源 |
|---|---|---|
| `dist_to_nearest_mrt_m` / `mrt_count_500m` | 最近捷運距離 / 500m 出入口數 | 捷運出入口座標(388 筆,cp950) |
| `bus_stops_count_300m` / `bus_stops_count_500m` | 300/500m 公車站數 | 公車站牌(3,407 筆) |
| `conv_stores_count_200m` / `conv_stores_count_500m` | 200/500m 超商數 | 超商門牌級座標(3,317 筆) |
| `restaurants_count_500m` | 500m 餐廳數 | 合法餐飲業(20,182 筆) |
| `dist_to_nearest_park_m` / `park_count_500m` | 最近公園距離 / 500m 公園數 | 公園基本資料(830 筆) |
| `dist_to_nearest_clinic_m` / `dist_to_nearest_school_m` | 最近診所 / 學校距離 | 診所 JSON / 中小學點位 |

**選用理由與誠實聲明**:POI 對**完整模型**增益 <1% AUC(訊號被鄰里特徵吸收),
但對**新房東冷啟動**情境有小幅增益(交通組 R² +8%),且為前端「機能快照」提供可信數據,
故全數保留。所有距離皆以 BallTree(haversine) 精確球面計算。

### 2.8 NLP 評論情緒(4 個)— 210,288 則評論萃取

| 特徵 | 繁中說明 | 選用理由 |
|---|---|---|
| `avg_review_sentiment` | 平均情感得分(0~1,正負向詞庫計分) | 住客真實口碑的量化;docx 功能模組「評論 NLP 情緒分析」 |
| `avg_review_length` | 評論平均字數 | 長評論通常代表深度體驗(正負皆然) |
| `has_no_reviews` | 無評論旗標 | 與 Y 單變數相關最高(+0.177);讓模型區分冷啟動房源 |
| `neg_review_ratio` | **負評比例**(含負向詞評論數 ÷ 總評論數) | docx To-Do「建立評論情緒與負評比例特徵」;中英負向詞庫(髒/吵/差/dirty/noisy/broken 等 21 詞)比對 |

### 2.9 明確排除的欄位(摘要)

除 §1 防洩漏欄位外,經實測排除:8 個照片品質特徵(亮度/對比/清晰度等,全量驗證訊號不穩)、
host_verifications / host_has_profile_pic / host_identity_verified(鑑別度極低)、
16 個設施旗標細分與標題長度(增益 <0.5% AUC,已被 amenities_count 吸收)。
**精簡優於堆疊**:實驗證明 11~13 個特徵即達誠實天花板,保留 59 個是為了沙盒操作性與冷啟動穩健性。

## 3. 使用的分析模型與選用理由

### 3.1 預測模型(雙軌 × 雙演算法 × 雙變體)

| 模型 | 角色 | 選用理由 | 誠實效能 |
|---|---|---|---|
| **LightGBM Regressor**(模型 A) | 預測連續空屋率 Ŷ | 梯度提升樹對中型表格資料最強;原生支援缺失值;訓練快(秒級),支撐沙盒即時 What-if | R² ≈ 0.24、MSE ≈ 0.072 |
| **LightGBM Classifier**(模型 B 主力) | 預測 P(Y≥0.6) | doc/01 實驗 10 組模型中 F1@0.6 最佳;葉序生長對本資料集擬合效率最好 | AUC 0.716 |
| **XGBoost Classifier**(模型 B 對照) | 同上,對照組 | 與 LGBM 同屬 GBDT 但生長策略不同;兩者 AUC 打平(0.715~0.720)**印證瓶頸在資料訊號而非模型**,側欄可即時切換對照 | AUC 0.720 |
| **Isotonic 校準**(CalibratedClassifierCV, cv=3) | 包裝兩個分類器 | GBDT 原始輸出非真實機率;等張迴歸校準後「機率 60%」才具有可對房東解釋的物理意義,60% 通知門檻才站得住 | 校準曲線貼近對角線 |
| **Logistic Regression / 線性迴歸** | 教學基準線 | docx To-Do 要求之 baseline;LogReg AUC 僅 0.65,證明 GBDT 的非線性容量必要 | AUC 0.652 |

**雙變體設計**:完整模型(59 特徵,老房東)/ 冷啟動模型(52 特徵,移除 7 個房東身分特徵,
新房東自動路由)。理由:新房東無歷史資料,若沿用完整模型會以中位數填補造成系統性偏誤;
冷啟動模型以地段/房間條件補位,對「沒見過的房東」表現更穩定(AUC 標準差更低)。

### 3.2 評估方法

| 方法 | 選用理由 |
|---|---|
| **GroupKFold(host_id, 5 折)** | 同房東多房源特徵高度相似,隨機切分會「房東洩漏」使分數虛高(單次切分 R² 可達 0.5+);依房東分組後測的是「面對全新房東」的真實泛化力,才是可對外出示的數字 |
| **雙層警報**(紅 P≥0.60 / 黃 P≥0.35) | 誠實 AUC 0.72 下 P/R 無法同時 ≥0.7;紅層 Precision≈0.70(通知寧缺勿濫)、黃層整體 Recall≈0.70(觀察名單漏抓最少),取兩者之長 |
| **OOF 全量預測輸出** | 總覽卡片/排名/通知中心顯示的全部是 OOF 誠實機率,不用樣本內擬合值(會虛胖) |

### 3.3 可解釋性模型

| 模型 | 用途 | 選用理由 |
|---|---|---|
| **LIME**(LimeTabularExplainer) | 房東端「原因 Top 3」 | docx 功能模組指定;局部線性近似輸出「規則式」解釋(如「近期平均最短入住 > 20.8 → +10.2pp」),對非技術房東最直觀 |
| **SHAP**(TreeExplainer) | 後台研究頁全域解釋 | 基於 Shapley 值,具可加性與一致性;蜂群/瀑布圖呈現全域特徵重要度與非線性型態,供研究與報告使用 |

> 兩者互補:LIME 快而直觀(面向房東)、SHAP 嚴謹而全面(面向分析者)。

### 3.4 建議生成

| 引擎 | 觸發條件 | 選用理由 |
|---|---|---|
| **LLM 智慧建議**(Gemini / Claude) | 🔴 紅色高風險(P≥0.60) | 將 LIME 原因 + 1km 跨平台競品落點 + 設施缺口組成提示詞,生成個人化、含具體數字的行動建議;高風險才觸發,控制 API 成本 |
| **規則引擎**(SuggestionEngine) | 黃/綠層、LLM 失敗後備、批次自動寄信 | 確定性、零成本、毫秒級;模板對應可控特徵(價格/天數/回覆/文案/設施缺口) |

### 3.5 跨平台競品比對(不進模型)

591(7,539 筆月租)、Booking.com(822 筆日租)、租租網 ddroom(543 筆月租)僅供
**1km 競品比較**(BallTree 索引、每人每晚等效價、同容量層落點、設施覆蓋率),
**不作為模型特徵**——決議理由:跨平台資料無歷史入住標籤,無法驗證因果;
作為對照面板提供房東決策脈絡,比硬塞進模型更誠實。

## 4. 一句話總結

以「未來一年空屋率」為真標籤(Y≥0.6 對齊 60% 通知),用 59 個涵蓋
**硬體、口碑、競爭、房東行為、用心度、地段 POI、評論情緒**的特徵,
訓練 LightGBM/XGBoost 雙模型 + Isotonic 校準,以 GroupKFold 誠實評估、
雙層警報運作;LIME/SHAP 負責解釋,高風險由 LLM 生成個人化建議,
跨平台資料只做競品比對不進模型。
