# 07 後台分析改造設計 — Airbnb 平台方視角

> Smart_accommodation_4 · 2026-07-22 設計定稿（brainstorming 決議）
> 範圍：改造 `pages/3_📊_後台分析.py`，使用者角色從「研究端/房東」轉為 **Airbnb 平台營運方**。

---

## 一、背景與定位

現有後台分析頁的主體是「策略沙盒」（選單一房東/房源、拉桿模擬空屋率），屬房東視角；
其餘 6 分頁（市場獲利地圖、評論面向分析、通知中心、模型評估、前瞻驗證、SHAP）各自獨立、缺乏平台方故事線。

本次改造將整頁重新定位為 **Airbnb 平台營運後台**，服務四大工作情境：

1. 監控市場健康度
2. 管理高風險房源／房東
3. 平台營收與成長分析
4. 服務品質監管

### 已決議事項

| 決議 | 內容 |
|---|---|
| 改造方式 | 改造現有 `3_📊_後台分析.py`，不另開新頁 |
| 策略沙盒 | **直接刪除**（房東入口已有 LIME／智慧建議，不搬移） |
| 營收口徑 | 預估年營收 = 房價 × (1−預估空屋率) × 365；**平台收入 = 營收 × 抽成率**（側欄滑桿，預設 15%，範圍 3%–20%） |
| 研究分頁 | 模型評估／前瞻驗證／SHAP **保留但合併收後排**（模型監控分頁） |
| 通知中心 | 獨立分頁移除，**整併進風險管理分頁** |
| 架構 | **五分頁**：市場總覽 / 風險管理 / 營收與成長 / 品質監管 / 模型監控 |

## 二、頁面架構

```
3_📊_後台分析.py（Airbnb 平台營運後台）
├── 側欄：全域篩選（行政區多選、房型多選）＋ 平台抽成率滑桿（3%–20%，預設 15%）
│         ※ 移除原「選房東／選房源」選單
├── 🏙 市場總覽        （新建）
├── 🚨 風險管理        （新建＋整合 notify_center）
├── 💰 營收與成長      （改造現有市場獲利地圖＋長短租策略）
├── 💬 品質監管        （現有 ABSA 分頁微調文案視角）

```

側欄篩選與抽成率透過 `st.session_state` 全分頁共用。

## 三、各分頁設計

### 3.1 🏙 市場總覽（新建）

**目的**：營運人員 10 秒掌握台北市場狀態。
**資料**：`data/_predictions.csv`（全房源 OOF 誠實預測）＋ `dataset_final.csv`，不重新訓練。

由上而下：

1. **KPI 卡片列（6 張）**
   - 總房源數、活躍房東數、全市平均預估空屋率、高風險房源占比（紅色警報層）、
     預估全市房東年營收總額、**預估 Airbnb 平台年收入**（× 抽成率）。
   - 抽成率滑桿拉動時全頁金額即時換算（簡報互動亮點）。
2. **行政區健康度總表**
   - 每區一列：房源數、平均空屋率、高風險占比、預估平台收入、與全市平均差異箭頭。
   - 欄位可排序；顏色深淺標出最健康 vs 最需關注行政區。
3. **風險分布地圖**
   - 重用 portfolio_sections 地圖能力，行政區著色高風險占比；旁附紅／黃／綠三層房源數長條。

### 3.2 🚨 風險管理（新建＋整合）

從「發現高風險」到「介入輔導」一條龍，內部三區塊：

**區塊 1：房源風險名單**
- 表格欄位：房源 ID、行政區、房型、房價、校準風險機率、警報層級（🔴≥0.60／🟡≥0.35／🟢）、預估年空屋率、房東 ID。
- 篩選：警報層級、行政區、房型、風險分數區間；預設風險機率降冪。
- 選取房源顯示 LIME Top-3 風險原因（重用 `modules/lime_explainer.py`）。

**區塊 2：房東層級彙總**
- 以 host_id 聚合：名下房源數、高風險間數、高風險占比、平均風險分數、名下預估總營收。
- 預設排序「高風險間數 × 占比」，優先找出整批房源惡化的房東。
- 點選房東下鑽名下房源風險明細。

**區塊 3：派發輔導通知**
- 名單勾選房源（或整個房東）→「產生輔導通知」→ 重用 `modules/notify_center.py`
  信件生成（LIME 原因＋ABSA 痛點＋改善建議）。
- 文案角度：由「房東收到警報」改為「Airbnb 平台主動關懷輔導」。
- 通知紀錄沿用現有 8 欄，概念上多「發送人：平台營運」。

### 3.3 💰 營收與成長（改造）

- 保留現有獲利地圖與長短租策略分析；所有金額同時顯示
  「房東端預估營收」與「平台端收入」（連動抽成率滑桿）。
- **新增「成長機會」區塊**：行政區 × 房型供需矩陣 —
  - 平均空屋率低（需求強）且房源數少（供給薄）→ **建議招募房源的缺口市場**；
  - 反之標示「已飽和、不宜再增供給」。

### 3.4 💬 品質監管（微調）

- 全市面向排行、四象限圖、行政區熱力圖、負評例句全部保留。
- 文案視角：由「市場痛點」改為「平台服務品質監管」
  （例：「建議平台對『隔音噪音』負評率前 10% 的房源發布品質改善要求」）。
- 加輕量跳轉：從痛點面向帶條件跳到風險管理名單；
  深度連動做不到時退而求其次做「複製房源 ID 清單」。

### 3.5 🧪 模型監控（三合一）

- 模型評估（LightGBM vs XGBoost）、前瞻驗證、SHAP 以內部子分頁收納，內容不改。
- 頂部加平台方視角說明：「本頁供平台方稽核 AI 風險評分的可信度」＋
  指標摘要卡（AUC 0.716、紅層 Precision 0.69、黃層 Recall 0.70）。

## 四、工程面

| 項目 | 決議 |
|---|---|
| 模型／資料檔 | 不重新訓練、不新增資料檔；新增計算全為 pandas 聚合（房東彙總、行政區彙總、供需矩陣） |
| 效能 | 聚合結果以 `@st.cache_data` 快取 |
| 新參數 | 平台抽成率（側欄滑桿，session_state 全頁共用），僅前端換算、不進模型 |
| 錯誤處理 | `_predictions.csv`／ABSA 快取缺失時，顯示「請先執行 scripts/train_backend_models.py」提示（沿用現有模式） |
| 影響範圍 | 主要動 `pages/3_📊_後台分析.py`；共用模組僅 `notify_center.py` 增加平台方文案分支（房東入口沿用原行為，不得破壞） |
| 驗證 | `streamlit run index.py` 逐分頁人工檢查；回歸確認房東入口／租客入口不受影響 |

## 五、不做的事（YAGNI）

- 介入成效模擬（平台級沙盒）— 明確不做。
- 單房源策略沙盒 — 刪除，不搬移。
- 真實金流／訂單資料 — 無此資料，一律以估算口徑並於 UI 標註「預估」。

### 實作期間的發現（與設計假設不同之處）

1. **房東入口已無通知中心**：設計時假設需保護房東端通知行為，實測 `render_notify_center` 目前唯一的
   live 呼叫者是新的風險管理分頁；房東入口分頁為 TB1/TB2P/TB5/TB2/TB6 共 5 個，通知中心在先前改版即已移除。
   `platform_view=False` 分支因此無 live 呼叫者，保留作為模組預設行為並由單元測試覆蓋。
   README「房東入口 … 通知中心」一段與現況不符，屬既有文件債，未在本次修改。
2. **部分房型房價缺失**：供需矩陣顯示中山區 Hotel room 中位價格為 0，代表該組合有大量 `price` 缺值，
   其預估營收會被低估為 0。營收類數字解讀時需留意此資料限制。
