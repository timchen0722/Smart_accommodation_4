## 四、使用的資料分析模型

### 4.1 既有模型(本次未更動)

| 模型 | 用途 | 演算法 | 誠實效能 |
|---|---|---|---|
| 模型 A | 預測連續空屋率 | LightGBM Regressor | OOF R² 0.243 / **前瞻 R² 0.003** |
| 模型 B(主力) | P(空屋率≥60%) | LightGBM + Isotonic 校準 | OOF AUC 0.716 / **前瞻 AUC 0.632** |
| 模型 B(對照) | 同上 | XGBoost + Isotonic 校準 | OOF AUC 0.720 |
| 解釋 | 單筆原因 | LIME(房東端)、SHAP(後台) | — |

### 4.2 本次新增的分析方法(刻意不訓練新模型)

| 方法 | 用途 | 為何不用機器學習 |
|---|---|---|
| **同儕分組實證統計**(peer benchmarking) | 營收最適定價、房型獲利、同商圈比較 | 前瞻驗證顯示模型外推 9 個月後 R²≈0;**用真實已訂天數的分組統計比模型預測更可信**,且房東容易理解(「同區同房型 17 間房的實際表現」) |
| **連續區段掃描**(run-length encoding) | 空檔警示 | 確定性演算法,無需學習 |
| **加權平均基準**(weighted market baseline) | 同商圈月度基準 | 樣本不足時自動退回同行政區,避免小樣本雜訊 |

> **設計原則**:能用真實資料直接算出來的,就不要用模型猜。
> 這也是前瞻驗證給我們的教訓——模型的長期外推能力有限,但**真實檔期資料本身就是答案**。

## 五、新增特徵欄位(中英對照與選用理由)

### 5.1 每房源檔期指標(`data/_calendar_metrics.csv`,共 30 欄)

| 英文欄位 | 繁體中文 | 型別 | 選用理由 |
|---|---|---|---|
| `listing_id` | 房源編號 | int | 與 listings 的 join key |
| `cal_start` | 日曆起始日 | date | 遮罩展開為日期序列的基準點 |
| `n_days` | 日曆天數 | int | 資料完整性檢查(正常為 365) |
| `booked_mask` | **訂房遮罩** | str(365 字元) | 核心欄位:每字元代表一天,`1`=不可訂、`0`=可訂。以字串壓縮取代 234 萬列明細,**記憶體占用降低約 97%**,是能在 Streamlit Cloud 運行的關鍵 |
| `booked_rate` | 全年已訂率 | float | 檔期健康度總指標 |
| `booked_days` | 全年已訂天數 | int | 營收估算的乘數(取代原本循環計算的 `estimated_occupancy`) |
| `booked_rate_d30` | 未來 30 天已訂率 | float | **近期經營壓力**:日曆已開放且訂單已成形,是最可信的訊號 |
| `booked_rate_d60` | 未來 60 天已訂率 | float | 中期檔期進度 |
| `booked_rate_d90` | 未來 90 天已訂率 | float | 主要判讀窗口(遠期資料受「尚未開放日曆」汙染) |
| `gap_count_90d` | 90 天內空檔段數 | int | 檔期破碎程度;段數多代表訂單零散難填 |
| `gap_days_90d` | 90 天內空檔總天數 | int | 可回收營收的規模 |
| `gap_longest_90d` | 最長連續空檔天數 | int | 決定促銷力度:≥21 天需大幅折扣或轉長租 |
| `gap_first_start` | 最近一段空檔起日 | date | 行動優先序:最快要處理的檔期 |
| `min_nights_median` | 最低入住天數中位 | float | 訂房門檻策略 |
| `min_nights_varies` | 最低天數是否逐日變動 | 0/1 | **市場實測 34% 房源已在做動態天數策略**,可作為經營成熟度指標 |
| `is_longterm_only` | 是否僅接長租 | 0/1 | 中位 ≥28 晚視為長租化(市場有 32.8 萬筆 30 晚設定,是法規因應訊號) |
| `is_all_blocked` | 全年封鎖旗標 | 0/1 | **資料陷阱防護**:260 間全年不可訂,多為停業/自住,須排除以免汙染統計 |
| `is_all_open` | 全年全空旗標 | 0/1 | 317 間完全無訂單,同樣排除 |
| `m1_rate` ~ `m12_rate` | 未來第 1~12 月已訂率 | float ×12 | 月度檔期進度,供與同商圈基準比較 |

### 5.2 市場基準表(`data/_calendar_market.csv`)

| 英文欄位 | 繁體中文 | 選用理由 |
|---|---|---|
| `neighbourhood_cleansed` | 行政區 | 商圈是需求的第一決定因素 |
| `room_type` | 房型 | 整棟/套房/雅房的需求結構不同,不可混比 |
| `mi` | 月份序號(1~12) | 相對月份,避免跨年比較錯位 |
| `mkt_rate` | 市場平均已訂率 | 同商圈同房型基準線 |
| `n_days` | 樣本天數 | 樣本不足(<3000 天)時自動退回同行政區基準 |

### 5.3 衍生分析欄位(即時計算,不落地)

| 中文名稱 | 計算方式 | 選用理由 |
|---|---|---|
| 年營收估算 | `每晚價格 × booked_days` | **關鍵改良**:已訂天數來自 calendar(獨立資料源),不再是 `doc/03 §3.3` 指出的 `price × occupancy` 循環恆等式 |
| 價格帶 | 同商圈同房型價格 8 分位 | 建立營收曲線的分組基礎 |
| 最適價格 | 營收曲線的極大值點 | 定價建議的依據 |
| 空檔可回收營收 | `空檔天數 × 每晚價格` | 把「12 天空檔」翻譯成「約 NT$X 的損失」,提高行動誘因 |
